01 — Problem

The gate optimizes for
the wrong trust contract

Affirm has one authentication gate for two fundamentally different situations: logging in (capability request) and paying a loan (obligation fulfillment). They are not the same. A user trying to pay is already highly motivated. Every friction point in that path deletes a signal worth more than almost any data model: a borrower trying to repay when we've made it hard.

The specific failure

The SSPC flow — the official recovery path — fails half the users who attempt it. For the users it does convert, the median resolution time is measured in days, not minutes. During that window, payments go unmade. The user's borrower record accumulates late signals. Support queue volume grows. None of this is caused by fraud risk; it's caused by a phone number changing. The system penalizes a user for a lifecycle event it could recover from cheaply.

02 — Approach

Identifier pairs, not passwords

The design question was: what's the minimum identity signal sufficient to authorize a single payment? Not account access — just payment. The answer is a pair of identifiers that can only both be known by the account owner, combined with proof-of-control of the email address on file.

Why not KBA?

Knowledge-Based Authentication was the first design. It was removed. KBA question corpora are largely solvable from data breach dumps. Email OTP provides actual proof-of-control — not just knowledge — and removes the need for question library maintenance and distractor generation entirely. It is strictly stronger.

Why fresh instrument only?

The session is authorized for one payment. Not for browsing, editing, or viewing account data. Surfacing stored cards would expand the authorization surface. The deliberate constraint — new instrument every time — is a security posture decision, not a product limitation.

Why a modal, not an inline state?

"That phone number is invalid" uses a full-screen overlay with fade-in + slide-up animation (24px → 0 travel). The modal gives two clear CTAs — update phone or pay without logging in — without competing with the sign-in form layout.

What was removed. What failed.

Removed: KBA flow
Knowledge-Based Authentication was the original plan for the second factor. Removed after realizing KBA corpora are largely solvable from public breach data. Email OTP is strictly stronger: it tests possession of an account, not knowledge of facts about it.
Removed: Card-on-file picker
Showing stored payment instruments in a session that bypasses normal auth would expand the authorization surface unnecessarily. Fresh instrument every time is a deliberate security posture, not a product gap.
Failed: Direct Resend API calls
Browser-side calls to api.resend.com fail due to CORS policy and sandbox restrictions. Fixed by adding a Vite dev proxy (/api/send-email → https://api.resend.com) so emails actually delivered to real inboxes during the demo.
Failed: MSW service worker scope
The mock service worker silently failed to intercept API calls — wrong base config meant the scope was off. Fixed by adding a Service-Worker-Allowed: / header and explicitly setting scope: '/'.

Key decisions

Email OTP over KBA
WHY Email OTP tests possession of an account identity, not knowledge of facts that may be compromised in public breach datasets.
TRADE-OFF Requires the user to have access to their email inbox. Users who've also lost email access need a different path (not in scope for M1).
Fresh instrument only, no card-on-file surfacing
WHY Authorization surface should match the trust level granted. The session grants one payment, not browsing capabilities. Surfacing stored instruments conflates the two.
TRADE-OFF More friction per payment, since the user can't tap Apple Pay from a saved card. Acceptable given the security constraint.
Modal for invalid phone, not inline error
WHY The modal gives the user two clear CTAs — update phone or pay without logging in — without competing visually with the sign-in form layout. Inline banners get buried.
TRADE-OFF Heavier interaction model. Acceptable because the modal represents a meaningful fork in the user's path, not a field correction.
Affirm DS tokens via CSS custom properties, not npm package
WHY @affirm/components-core requires Artifactory credentials and has Vite compatibility issues. Token values were lifted from DS Core Figma and mapped to CSS custom properties manually.
TRADE-OFF Drift risk from Figma source. Acceptable for a prototype. Production would use the real package.
04 — Trust Pulse (AI layer)

Risk scoring before
the OTP step

Trust Pulse is a standalone Python service (FastAPI + LangChain + Claude Sonnet) that scores each request before the authentication step. It advises, it does not decide. The risk_framework makes the final call. This is a deliberate constraint: AI output is advisory, not authoritative.

The 500× cost argument

Biometric IDV trades accuracy for cost. Trust Pulse doesn't replace Onfido for high-risk cases — it fast-lanes the 70–80% of genuinely low-risk requests so expensive checks aren't wasted on them. The comparison is about the cost per correctly-routed low-risk user, not head-to-head accuracy.

Architecture constraint: AI advises, rules decide
The score-to-tier mapping is enforced in server code, not trusted from LLM output. Belt-and-suspenders: the LLM returns a score and tier, the server re-derives the tier from the score using the defined thresholds. An adversarially-crafted LLM response cannot self-authorize a payment by claiming tier=low with a score of 0.9.

06 — How we think

Four sharp statements

Authorization surface

The authorization granted should match the trust level established — not the maximum possible. A session that proves "this person can receive email at the address on the account" authorizes payment. It does not authorize browsing stored cards, editing account details, or any other capability. Conflating trust levels creates risk; distinguishing them is just engineering discipline.

Friction as signal

Payment intent in a hard-to-pay context is a strong signal. A user who finds a rescue flow, navigates it correctly, and completes an OTP step is almost certainly not a fraudster — fraudsters target easy paths. Optimizing that flow for speed is not a security trade-off; it's correctly reading what the friction has already filtered.

AI as advisor, not authority

Trust Pulse scores requests. It does not authorize them. The score-to-tier mapping is enforced in code independent of LLM output. This isn't distrust of the model — it's recognizing that a system where adversarial input to the LLM can change an auth decision is not a security system. The model advises; the rules decide.

Resilience is a product feature

Trust Pulse has a 1.5-second hard timeout. On any failure — timeout, 5xx, malformed response — it defaults to tier=med and standard OTP. The user never encounters a degraded state. Cached verdicts serve pre-scored demo scenarios in under 100ms. A feature that degrades visibly under failure is only conditionally shipping. The fallback is part of the design.
